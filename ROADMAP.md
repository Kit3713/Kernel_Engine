# Roadmap — Ironclad

Development is organized into phased milestones from pre-alpha through stable release. Versions follow semantic versioning: alpha releases span 0.1.x through 0.5.x, beta begins at 0.6.x, and the first stable release is 1.0. Early phases prioritize a correct compiler and a single working end-to-end path — Ironclad source to a bootable system built from a certified ISO — before broadening the standard library or introducing fleet-scale topology.

The build model is toolchain-based: the compiler emits ordered scripts and configuration files that orchestrate certified tools (`parted`, `cryptsetup`, `dnf`, `semodule`, `nft`, Kickstart, etc.) to realize the declared system. Where no certified tool covers an operation, the compiler generates bash. The less Ironclad reimplements, the smaller the codebase and the greater the trust inherited from the underlying platform.

---

## Phase 1 — Core Parser and Compiler Skeleton (0.1.x)

Define and formalize the Ironclad grammar using pest (PEG). Implement the core parser: tokenization, AST construction, and syntax validation. The grammar covers the language's core primitives — files, directories, symlinks, mount points, packages, users, groups, permissions, ownership, SELinux labels, mutability flags — along with variables, loops, conditionals, and the single-inheritance class system. The grammar also covers the six compiler-native subsystem domains: storage topology, filesystem tree, SELinux declarations, service/init blocks, firewall rules, and network interfaces. Implement the class resolution pass: inheritance flattening, property resolution, variable substitution, and origin tracking. Produce structured compiler output: diagnostic logs, the resolved AST, and a human-readable representation of the flattened system declaration. No backend emission at this stage. The objective is a correct, well-tested parser and resolver that accepts valid Ironclad source and rejects invalid input with precise, actionable error reporting.

## Phase 2 — Toolchain Emission and Proof of Concept (0.2.x)

Implement the intermediate manifest: CBOR serialization of the resolved AST, Ed25519 signing, and manifest verification. Implement the toolchain emitter: translate the resolved AST into an ordered set of scripts and configuration files that orchestrate certified tools to build the declared system. The primary build target is `iso` — starting from a certified minimal ISO (Fedora or AlmaLinux), the toolchain uses Kickstart for disk partitioning and initial package installation (preserving the ISO's certification chain), then orchestrates `useradd`, `chown`, `chmod`, `nft`, `semodule`, `systemctl`, and bash for all subsequent configuration. Implement the `chroot` target as a secondary path for development and testing. Validate the end-to-end pipeline: Ironclad source → compiler → toolchain → bootable installed system with LUKS2 encryption, SELinux enforcing, and a signed manifest on disk. Ship the initial standard library classes: `HardenedRHELBase` (minimal hardened server), a basic service configuration class, and a basic firewall policy class. Standard library classes are written in Ironclad using only the language's core primitives.

## Phase 3 — Semantic Validation and Cross-Domain Checking (0.3.x)

Implement the full semantic validation pass across all six compiler-native domains. Structural checks: conflicting path declarations, files on undeclared mount points, mutable files on read-only filesystems, security floor enforcement (SELinux enforcing, LUKS2, immutable root). Cross-domain checks: services reference declared users and SELinux types, firewall rules reference declared network interfaces, service ports have corresponding firewall rules (and vice versa), network interface addresses are unique, package references are satisfied. Expand the standard library with subsystem classes for s6 supervision, sshd configuration, chrony, rsyslog, and user/group management helpers. Ship `S6ContainerHost`, `SystemdServer`, and `SecureDesktop` base classes. Implement inheritance depth warnings and compiler ergonomics (better error messages, source location tracking through inheritance chains).

## Phase 4 — SELinux Targeted Policy Backend (0.4.x)

Implement the SELinux targeted policy backend in the compiler: analyze the fully resolved AST to generate correct `.te`, `.fc`, and `.if` policy modules using the Reference Policy as a foundation. The compiler uses its global view of the declared system topology — services, files, users, network interfaces, and their labels — to emit policy that no single class could generate from local context. Implement strictness parameterization: a single compiler flag producing policies of graduated restrictiveness. Implement manual override handling: preserve engineer-authored policy files across recompilation, flag conflicts when declarations change. Validate generated policy against real systems — this phase requires extensive integration testing. Engineers who prefer to author SELinux policy entirely by hand can declare their policy files through file primitives; the compiler backend is an accelerator, not a requirement.

## Phase 5 — Workload Classes and Build Target Expansion (0.5.x)

Implement standard library classes for workload declaration. Podman container class: accepts image references, network bindings, volume mounts, resource limits, and restart policy; emits Quadlet `.container` files integrated with the declared init system. Libvirt VM class: accepts resource allocations, network attachments, firmware type, and boot configuration; emits domain XML. Extend semantic validation to cover cross-declaration conflicts surfaced by workload classes: port collisions between host services and containers, volume mounts referencing undeclared paths, VM resource demands exceeding declared host capacity. Implement the `image` build target (bootc Containerfile emission for OCI container images) and the `bare` build target (full toolchain for bare disk, Linux From Scratch-style builds, and embedded devices). Ship `ContainerHost`, `VMHost`, and `EmbeddedAppliance` base classes.

## Phase 6 — Runtime Agent and Drift Detection (0.6.x)

Implement the runtime agent in Rust: manifest reading and signature verification, live state comparison against all declared filesystem properties (content hashes, permissions, ownership, SELinux labels), structured drift reporting to configurable sinks (local file, syslog, remote endpoint). Install the agent via the build toolchain on every Ironclad-built system. The agent performs detection and reporting only — no remediation. Implement delta toolchain generation: given an old manifest and a new declaration, the compiler emits the minimum set of scripts to move from the old state to the new. Implement post-maintenance verification: triggered comparison after delta application, convergence reporting. Extend the manifest to include workload state (expected running containers, expected VM states) for classes that declare them. Ship the agent as a statically-linked binary with minimal dependencies.

## Phase 7 — Topology, Fleet Composition, and Kubernetes (0.7.x / Beta)

Implement topology declarations: systems as first-class values, variable assignment, composition of multiple system declarations into fleet-level descriptions. Implement cross-system semantic validation: network reference consistency, IP address uniqueness, aggregate resource validation. Implement per-system manifest generation within topologies and per-system toolchain emission. Ship Kubernetes standard library classes: kubeadm configuration, CNI plugin manifests, kubelet service integration, with control plane and worker node classes that compose into cluster topologies. Validate the full lifecycle at scale: initial fleet build from declaration, runtime drift detection across the fleet, delta redeployment when the declaration changes. Harden the compiler and agent against edge cases and adversarial input.

## Full Release (1.0)

Production-grade compiler with all validation passes and all build targets (`iso`, `chroot`, `image`, `bare`, `delta`). Runtime agent stable for production. Comprehensive standard library: system base classes for servers, desktops, embedded devices, and container hosts; subsystem classes for all compiler-native domains plus bootloader, secrets management, and common application stacks; topology composition classes for clusters and fleets. Complete documentation: language reference, grammar specification, class authoring guide, standard library reference, runtime agent configuration, and topology guide. Production-grade for Fedora and AlmaLinux targets with desktop and embedded profiles alongside server profiles.

## Post-1.0

SELinux MLS policy compiler backend, extending the targeted policy architecture with sensitivity levels, categories, and formal information flow verification. Additional distribution targets (Debian, Arch, Alpine) requiring new standard library base classes and package management adaptations. Community class repository for contributed and shared Ironclad classes. Fleet topology extensions: deployment orchestration, rolling updates across topologies, cross-system drift correlation. Cloud-init compatibility classes for cloud deployments. Terraform provider for infrastructure-as-code integration. RHEL proper support. Expanded embedded and IoT profiles. Desktop environment classes (GNOME, KDE, Sway) with security-hardened defaults.

---

Early phases prioritize a correct compiler and a single working path from source to bootable system. The first milestone that produces a real, bootable system from Ironclad source — Phase 2 — is the project's proof of concept. Everything after it builds on that foundation. The build model at every phase is the same: the compiler orchestrates certified tools with validated inputs, and bash fills the gaps.
